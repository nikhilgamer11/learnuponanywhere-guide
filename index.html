The location determines the iFrame approach, SSO requirements, and what the learner experience will feel like. Be as specific as possible.

Say this
"Before I scope this, can you tell me how your app is built — is it native Swift or Kotlin, or does it use a wrapper like React Native or Flutter?"

B — Mobile Architecture
How is their platform or mobile app built?
This is the most important technical question in any Anywhere deal. Get this answer before you demo anything on mobile.
High riskFully native app — Anywhere is not natively supported
Risk
LUA loads into an iFrame (wrapper). Native apps built in Swift or Kotlin do not have a WebView by default. This requires significant dev work and has a high factor of unknowns.
Say this
"LearnUpon Anywhere loads inside a WebView — a browser window within the app. If your app is fully native today, your dev team would need to add WebView support. Is that something they could explore?"
Do not
Do not demo mobile Anywhere or commit to a mobile test until the Webview question is resolved.
Alternative
Offer a browser-based version of Anywhere as an interim option — mobile browser is fully supported with no limitations.
Further discovery
"Is there any plan to add WebView capability to the app architecture in the near term?"
"Who from your dev team would own the Webview implementation? How long would that take?"
"Would a mobile browser experience be acceptable as a starting point while the Webview work is scoped?"
Proceed with careHybrid app — works well but two known limitations apply
Limitations
Video/audio file downloads are blocked by default. Assignment recording is blocked by mobile permissions. Both are fixable with a short code change.
Ask
"Do your learners need to download video or audio for offline use? And will you use assignment modules that require video or audio recording?"
Note
Large dependency on developer competency. Confirm dev resource is available before committing to a test timeline.
Must askApp architecture not confirmed — this is your first question on the call
Why
This single answer changes the entire scoping conversation. Do not proceed with a mobile demo or test proposal until you have this answer.
Ask first
"Before we go any further — how is your app built? Is it native Swift or Kotlin, or does it use a wrapper framework like React Native or Flutter?"
Green signalBrowser-based — fully supported, no limitations
Status
Desktop and mobile browser is fully supported. All module types work. SCORM/xAPI works. White-labelled portals supported. Best starting point for most deals.
Note
LUA is the most lightweight and easiest to implement headless LMS solution — more so than Absorb or Docebo.
Is mobile-first a critical value driver for this deal?
"Mobile-first" means different things to different prospects. The answer here determines how much the app architecture risk actually matters.
High riskMobile-critical + architecture unconfirmed — validate before progressing
Risk
If mobile is the core requirement and they have a native app, this deal has significant implementation risk. Do not let this progress to demo without confirming the app architecture.
Ask
"When you say mobile-first — is mobile the only access method, or would desktop browser be acceptable as a starting point during the test?"
Ask
"What percentage of your learners would access training via mobile versus desktop today?"
C — Content and Media
What content types are they planning to use in Anywhere?
Some content types have known limitations in embedded environments — particularly on mobile. Select the most complex type they need.
Action requiredDispatched SCORM — content vault must be enabled first
Risk
SCORM dispatched from an external tool (e.g. Articulate, Adobe Captivate) needs Content Vault enabled in the LU portal settings. Missing this step breaks content in the embedded view.
Action
Confirm Content Vault is enabled before scoping the test. Quick fix but it cannot be skipped.
Ask
"What tool are you using to create and host your SCORM content? Is it dispatched from Articulate or another platform, or uploaded directly into LearnUpon?"
Technical flagAssignment recording blocked on mobile by default
Risk
Video and audio recording in Assignment modules is blocked by default on hybrid and native mobile apps due to iFrame permission restrictions.
Fix
Fixable with a short iFrame permission code change by their dev team. Needs to be flagged before the test is scoped.
Ask
"Is assignment recording critical to the first use case, or could it be excluded from the initial test?"
Ask
"Does your dev team have capacity to apply a short iFrame permission update as part of the test setup?"
Not supportedOffline download not available on hybrid or native apps
Risk
Video and audio file downloads are blocked on hybrid and native mobile apps. Content can be streamed but not downloaded for offline use.
Say this
"Offline download is not supported in the embedded experience on mobile apps. Streaming works fine — but if offline download is a hard requirement, we need to discuss alternatives."
Ask
"Is offline download a hard requirement, or is it a nice-to-have? Most learners with a mobile connection find streaming acceptable."
Are they embedding full courses or specific modules only?
LUA is built for learner experience — it supports full courses. Module-only embedding is not supported.
LimitationModule-only embedding is not supported in LUA
Risk
LearnUpon Anywhere is built for learner experience and supports full courses only. Individual module-only embedding is not available.
Say this
"LUA embeds at the course level — so the learner accesses a full course within your platform. If you need a single standalone module, we would need to wrap it in a course first."
Ask
"Would wrapping the content in a short course work for your use case, or is standalone module access a hard requirement?"
D — Technical Setup
Do they have Single Sign-On (SSO) configured?
SSO determines how seamless the learner experience will be. No SSO means friction and a longer test setup. SAML and SQSSO are both supported.
BlockerNo SSO — test setup will be significantly longer
Risk
Without SSO, learners cannot access the embedded experience seamlessly. The test can still be done but the setup is more complex and the learner experience will have friction.
Ask
"Is SSO on your roadmap? Even a basic SQSSO implementation would significantly simplify the test setup and the final learner experience."
Note
SQSSO is the simplest option — it can be set up quickly without a full IDP. Worth introducing this as the path of least resistance.
Further discovery
"How do your users currently log in to the platform where learning will appear?"
"Is there a plan to implement SSO in the near term? What IDP would you use?"
"Would SQSSO — a simpler LearnUpon-native SSO option — be acceptable as a starting point?"
Technical checkCookieless authentication — IDP compatibility must be confirmed
Risk
Cookieless implementations are advised for embedded experiences, however SAML + Cookieless can break depending on the IDP provider.
Action
Confirm IDP provider and test cookieless compatibility as part of the technical validation call before scoping the test.
ValidateSSO exists but needs validation against LearnUpon
Action
Schedule a 30-minute technical call with their IT contact to validate the SSO connection before the test starts. This is a quick step but cannot be skipped.
Ask
"Which IDP provider are you using — Okta, Azure AD, Google, or something else? And is it SAML-based?"
Do they have development resource available?
No dev resource means Grab & Go is the only viable option. Scoping a Custom iFrame without a developer is the most common mistake in Anywhere deals.
Scope changeNo dev resource — redirect to Grab & Go immediately
Action
Do not scope a Custom iFrame deal. Grab & Go is the only realistic option — LearnUpon provides a code snippet, no development required.
Say this
"If you do not have dev resource right now, we have a Grab & Go option — LearnUpon provides a code snippet you paste into any page. It is the fastest way to get started without engineering involvement."
Note
Grab & Go is static content — no personalisation or dynamic user data. Set that expectation clearly.
Qualify furtherLimited dev resource — name a technical contact before scoping
Risk
Without a named technical owner, the test will stall when it reaches the integration phase.
Ask
"Who specifically from your technical team would own the integration side? Even part-time — I need a name before we set a test start date."
Ask
"What is their current backlog situation? Can we get two to three days of their time in the next two weeks?"
Must confirmTechnical contact not identified — get a name before you leave the call
Ask on call
"Who from your technical team would own the integration side of the test? I need to send them the Getting Started documentation directly."
